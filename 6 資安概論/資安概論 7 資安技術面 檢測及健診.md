- 弱點掃描
	- 自動化工具 與 非侵入式檢查
	- 識別其中已知或潛在的安全漏洞，例如軟體版本過舊、不安全組態設定、預設密碼、開啟不必要的服務埠，以及已公開的軟體漏洞等
	- 目的
		- 發現潛在弱點
		- 提供改善建議
		- 作為管理依據
		- 確認弱點已排除
		- 提升系統安全
	- 弱點掃描類型
		- 主機系統弱點掃描
			- 檢查其作業系統、網路服務、系統設定、帳號密碼管理等
			- 至少應符合 共同脆弱性及暴露 (Common Vulnerabilities and Exposures, CVE) 發布的弱點內容
				- 作業系統未修正的弱點
				- 常用應用程式的弱點
				- 網路服務程式的弱點
				- 網路通訊埠掃描
				- 木馬或後門程式的掃描
				- 帳號密碼破解測試
				- 系統的不安全或錯誤設定
		- Web網頁弱點掃描
			- 符合OWASP TOP 10：2021等最新版本
				- A01 Broken Access Control（權限控制失效）
				- A02 Cryptographic Failures（加密失敗）
				- A03 Injection（注入式攻擊）
				- A04 Insecure Design（不安全設計）
				- A05 Security Misconfiguration（安全設定錯誤）
				- A06 Vulnerable and Outdated Components( 脆弱及過時的組件）
				- A07 Identification and Authentication Failures（身分識別及鑑別失效）
				- A08 Software and Data Integrity Failures（軟體及資料完整性失效）
				- A09 Security Logging and Monitoring Failures（安全日誌及監控失敗）
				- A10 Server-Side Request Forgery（伺服器端請求偽造）
	- 弱點掃描流程
		- 確認掃描需求(Plan)
			- 定義掃描 目標、類型、範圍，確認 合規性要求及排程
		- 執行初掃(Do)
			- 選定工具（如 Nessus, Qualys)，並配置 掃描參數，執行 自動化掃描
		- 分析初掃報告(Check)
			- 排除誤報，並依據 風險等級、影響程度 對弱點進行優先級排序
		- 確認並修復弱點(Act)
			- 對高風險弱點進行驗證
			- 執行修補（安裝修補程式、修正組態、修正程式碼）
			- 最重要的環節，強調修復的「執行力」
		- 執行複測(Do)
		- 生成複測報告(Check)
	- 如何執行弱點掃描
		- 事前準備與了解
			- 了解其網路架構、系統環境（設備廠牌、系統版本等）
		- 工具準備
			- 合法授權的商業掃描軟體
			- 弱點資料庫更新至最新版本
		- 安排適當的掃描時間（通常在非公務時段或與組織
	- 弱點掃描報告
		- 執行結果摘要
		- 弱點統計分析
		- 詳細弱點清單
			- 提供具體的修補建議
		- 掃描誤判清單
			- 誤判的項目及理由
		- 弱點排除清單
			- 特定原因（如無法修補、已有其他配套措施）而暫不修補的弱點及其理由
		- 複掃報告的差異化分析
	- 弱點掃描考慮因素
		- 內部掃描 VS 外部掃描
		- 有登錄權限(Authenticated) VS 無登錄權限(Unauthenticated)
		- 侵入式 VS 非侵入式
		- 委外 VS 機關IT 資安部門
		- 內部掃描 VS 外部掃描
			- 外部掃描
				- 模擬駭客從外部進行的 資訊收集 及 初步攻擊嘗試
				- 發起位置在組織的防火牆「外部」
				- 掃描流量 從外部通過GSN 政府網際網路 及Firewall
				- 目標：主要檢測組織對外開放的服務
					- Web Server（網站伺服器）
						- 公網IP位址 及 相關的Web應用程式漏洞
					- Mail Server（郵件伺服器）
						- SMTP/POP3/IMAP服務漏洞
					- FTP Server（檔案傳輸伺服器）
						- 如果對外開放
					- 防火牆的規則組態是否安全、是否有不必要的埠口開放
			- 內部掃描
				- 模擬內部人員的惡意行為、內部電腦被入侵後作為跳板的行為 或 內部組態錯誤導致的風險
				- 發起位置在組織的防火牆「內部」 也就是內部區域網路 (LAN) 中
				- 目標：檢測組織內部網路中的所有資產
					- 內部使用的 Web Server、Mail Server、FTP Server（即使不對外開放）
					- 內部業務系統的漏洞，例如資料庫伺服器、ERP/CRM 系統等
					- 內網的Switches（交換器）等網路設備的組態弱點
					- 各個 Client端電腦（工作站）的作業系統、應用程式、組態漏洞
		- 有登錄權限(Authenticated) VS 無登錄權限(Unauthenticated)
			- 無登錄權限掃描
				- 模擬一個完全陌生的外部駭客進行初步探測
				- 只能測試「無需登入」即可公開存取的服務及介面
					- 服務版本漏洞、首頁上可觸發的XSS 漏洞、開放埠等
					- 無法探測登入後的功能，容易遺漏深層次的應用程式邏輯漏洞
			- 有登錄權限掃描
				- 模擬內部人員的惡意行為，或外部攻擊者成功入侵後進行的內部探測
				- 模擬合法用戶行為，能深入檢查系統內部配置 及 應用程式的登入後功能
					- 作業系統組態設定、內部服務漏洞、權限問題（水平/垂直）、登入後表單的注入攻擊、敏感資料存取等
						- 水平越權是看了同一層級別人的資料
						- 垂直越權是做了更高層級才能做的事
					- 觸及更多程式碼路徑，結果更全面、準確，誤報率較低
		- 侵入式 VS 非侵入式
			- 非侵入式
				- 不會對目標系統造成影響，主要透過分析響應判斷弱點
					- 不會嘗試修改系統的組態、資料或運行狀態
					- 不會利用漏洞進行實際的攻擊
					- 透過向 目標系統 發送 特定的請求（如端口掃描、服務指紋識別、Banner Grabbing 等），然後 分析系統反應 判斷弱點
						- 端口掃描
							- 檢查目標主機有哪些網路埠是開放的
								- 22 埠開放：可能是 SSH
								- 80 埠開放：可能是 HTTP 網站
								- 443 埠開放：可能是 HTTPS 網站
								- 3389 埠開放：可能是 Windows 遠端桌面
						- 服務指紋識別
							- 對開放的端口進一步發送特定請求，分析回應內容、格式、行為或通訊特徵，以判斷
								- 執行的是哪種服務
								- 使用什麼軟體版本
								- 可能的作業系統
						- Banner Grabbing
							- 有些服務在連線後，會主動回傳自己的名稱、版本或歡迎訊息，這些資訊稱為 Banner
				- 安全性高，適合運作環境
			- 侵入式
				- 可能對系統造成不穩定或破壞，通常用於滲透測試，需謹慎授權
					- 執行某些操作（如注入惡意程式碼、提升權限、獲取敏感資料等）
					- 通常在 測試環境 而 非生產環境 中進行
				- 弱點掃描主要採用非侵入式

- 滲透測試
	- 針對 權限跳脫 與 邏輯錯誤等深層問題 進行模擬攻擊
	- 滲透測試之類型及類別
		- 測試類型
			- 作業系統
				- 遠端服務、本機服務
			- 網站服務
				- 設定管理、使用者認證、連線管理、使用者授權、邏輯漏洞、輸入驗證、Web Service、Ajax
			- 應用程式
				- 電子郵件服務套件、檔案傳檔服務套件、遠端連線服務套件、網路服務套件、其它
			- 密碼破解
				- 密碼強度測試
			- 無線服務
				- 無線服務弱點測試

- 應用程式安全
	- SSDLC
		- 需求分析(Requirements)
			- 資安需求
		- 架構設計(Design)
			- 資安架構
		- 程式實作(Implementation)
			- 正確安全的程式撰寫習慣
		- 測試與驗收(Testing)
			- 依據資安需求擬訂測試計畫
		- 部署與維運(Maintenance)
			- 定期修補漏洞(Patch)、按步升級更新版本(Upgrade) 及即時監控(Monitor)
	- 安全控制- 變更控制
		- 原因
			- 需求變更、新功能要求 及 發現瑕疵等
			- 變更是常態
			- 風險管理
		- 方法
			- 實作標準變更控制流程
				- 流程化
					- 從變更「提出、評估、規劃、開發、測試、部署到驗證」的每一步都有明確規範
				- 協作
					- 涉及跨部門（業務、開發、測試、資安）的協同作業
				- 版本控制
					- 利用工具（如 Git)追蹤程式碼變更歷史
			- 確保變更的三個關鍵要素
				- 授權
				- 測試
				- 記錄
		- 步驟
			- 變更需求管理
			- 變更評估
			- 變更審查
			- 變更測試與部署
			- 變更結果報告
	- 安全控制- 職責區隔
		- 目的在於防止單一人員在 軟體開發與部署過程中 濫用權限，或 無意中造成潛在的安全漏洞
		- 作業人員 不應 有權限存取線上的程式碼或程式物件
		- 程式設計人員 不應 存取線上運作中的軟體
		- 品管部門 應測試程式碼品質，且與 開發部門 採用不同的測試方法
		- 一旦軟體被開發測試完成 應被保存在程式庫中
		- 線上運作的軟體 應由 程式庫中發行，不應直接由 程式設計人員或測試人員 進行更新
		- 開發、測試、上線等角色分工明確，相互獨立
	- 安全控制- 程式庫維護
		- 集中存放 與 存取控管
		- 版本控制
		- 開發與測試流程整合
			- 凍結版本後，應簽入 (Check In) 到程式庫 (Check Out) 取得最新版本進行修改、測試
		- 上線發行
			- 應由程式庫中發行(Release) 最新版本應用程式
	- 行動應用程式安全
		- 開發注意事項
			- 採用安全第三方套件
			- 避免索取過多敏感資訊
				- 如通訊錄、行事曆、座標位置、郵件、簡訊內容等
		- 使用者注意事項
			- 不要 在手機裡儲存重要資料
			- 不要 下載不明APP
	- 安全威脅 - 輸入攻擊
		- SQL 注入 (SQL Injection) 及跨站腳本 (Cross-Site Scripting, XSS)
			- 未能對 使用者輸入 進行充分檢查與過濾的弱點
			- 導致程式錯誤、執行邏輯被改變，甚至造成存取權限的跳脫
	- WEB 應用程式 - 檢測
		- 檢測頻率
			- A級機關每年 2次弱點掃描 及 1次滲透測試
			- B 級機關每年 1 次弱點掃描 及 每 2 年 1 次滲透測試
		- 若預算充足，可同時採用 自動源碼掃描 與 人工滲透測試
		- 若預算有限，則優先採用 自動源碼掃描

			- 基本環境調查

						- 就是把交換器某個連接埠的網路流量，**複製一份到另一個指定連接埠**，讓資安或網管設備進行監看與分析
			- 決定範圍與抽樣方式
				- 範圍與抽樣方式 須具有代表性
			- 檢測配合事項
			- 啟始會議
				- 說明檢測 項目與範圍，協調 配合事項
				- 參與人員 應包含 機關主管與相關業務承辦人
			- 執行檢測
			- 撰寫檢測報告
			- 提出改善建議
			- 結束會議
				- 說明 各項目發現事項、改善建議與結論
				- 參與人員 應包含 機關主管與相關業務承辦人
			- 修補規劃與追蹤
				- 優先處理 可即時修補 與 風險等級較高的弱點
				- 無法即時修補之弱點
					- 規劃改善計畫與改善時程
					- 持續追蹤修補進度
				- 執行複測，以確認修補方法之有效性

- 網路安全
	- 網路區域規劃
		- 避免來自內部與外部的攻擊，並有效防範災害擴大
		- 區域劃分項目包括
			- 外部網路（WAN）
			- 內部網路(LAN)
			- 非軍事區(DMZ)
			- 網路管理區域
			- 網路記錄區域
			- 實體隔離區域
		- 外部網路（WAN）
			- 機關對外網路區域，連接外部廣域網路(WAN)
			- 對內部 需要經過 防火牆的存取控制
			- 非允許的 服務與來源 不能進入其他區域
				- 防火牆的規則 會確保 只有經過明確允許的服務(例如特定的通訊埠)及來自特定來源的連線 才能通過
		- 內部網路(LAN)
			- 內部網路通常包含機關使用者終端、資通系統等設備
			- 利用VLAN與不同路由等方式切開
				- 虛擬區域網路(VLAN， Virtual Local Area Network)
				- 不同 VLAN 之間的通訊通常需要透過 路由器或交換機(可在此處設定安全策略)
			- 外部網路無法直接連線內部網路
				- 防火牆的規則
			- 並可考慮於 內部網路 架設代理伺服器 控管與 外部網段之連線
				- 所有內部使用者 對 外部的網路 請求都會先經過 代理伺服器
				- 內容過濾、流量監控、身分驗證等
		- 非軍事區(DMZ)
			- 主要用於放置機關對外服務伺服器(如網頁伺服器、FTP伺服器)
				- 網頁伺服器 : 提供 網站服務 給外部使用者
				- FTP 伺服器 : 提供 檔案傳輸服務 給外部使用者或合作夥伴
				- 其他如郵件伺服器、DNS 伺服器、VPN 伺服器等也常部署在 DMZ
			- 僅開放特定服務
				- 對 外部網路 只會開放 其提供服務所需的 特定通訊埠
					- 如網頁伺服器通常只開放 TCP 80 (HTTP) 及 TCP 443 (HTTPS)
				- 「最小權限」原則可以限制攻擊面
			- 需要嚴密控管此區域到 內部區域 的存取
				- 通常會有第二道防火牆或更嚴格的防火牆規則
		- 網路管理區域
			- 放置機關網路管理設備(如網路管理系統、鑑別系統、監控系統)
				- 網路管理系統 (NMS)
					- 用於監控、組態及管理網路設備(如路由器、交換機、防火牆等)
				- 鑑別系統
					- 如 RADIUS(Remote Authentication Dial-In User Service) 或 TACACS+(Terminal Access Controller Access-Control System Plus) 伺服器 
						- **RADIUS**：較常用於一般使用者連接網路，例如 Wi-Fi、VPN、802.1X
						- **TACACS+**：較適合管理員登入路由器、交換器、防火牆，因為可以細緻控制「每一條管理指令」的權限。
					- 對 管理員 登入網路設備 進行 身分鑑別及授權
					- **AAA**：
						- Authentication 身分鑑別＋Authorization 授權＋Accounting 稽核紀錄
				- 監控系統
					- 收集 網路效能數據、日誌，並提供告警功能
			- 應明確標示 網路的路徑及維運 方式
				- 其 網路連接路徑(如哪些 管理設備 連接到哪些 被管理設備)，以及維護操作的標準程序，都應該被清晰地定義及記錄
			- 網路設備維運 應該與 服務的網段 有所區隔
				- 非常重要的安全原則
				- 管理網路設備的連線(例如 SSH、Telnet、SNMP 等) 應該與 提供業務服務的網路流量 分開
				- 應 使用獨立的管理網段 (Out-of-Band Management) 或 至少是邏輯上隔離的 VLAN
				- 避免當 服務網段被攻擊，進而影響 網路設備的管理
		- 網路記錄區域
			- 放置 備份主機(Backup Server)及 記錄主機(Log Server)
				- 備份主機 (Backup Server)
					- 儲存 各類系統、資料庫、應用程式 等的備份
					- 備份主機 放置在 一個獨立且受保護的區域
				- 記錄主機 (Log Server)
					- 集中收集來自網路設備、伺服器、應用程式等 各種系統的日誌 (log) 資訊
					- 對於安全事件的 監控、分析、溯源及稽核至關重要
			- 存放的網段應特別規劃
				- 與其他業務網段、使用者網段等 進行嚴格的邏輯或實體隔離
			- 僅允許 設備備份 與 記錄之發送 及 管理員管理連線行為
		- 實體隔離區域
			- 提供最高等級的隔離
			- 機關依據特定需求可將 部分區域 執行實體隔離
				- 與 其他網路間 完全沒有實質的網路連線
				- 處理 國家機密的系統、研發核心技術的實驗室網路、關鍵基礎設施系統
			- 單向傳輸
				- 只能從 高安全區域 傳輸到 低安全區域
				- 透過 光閘 (data diode) 等單向資料傳輸設備 實現
				- 防止 外部惡意病毒或攻擊 進入
			- 可能危害途徑
				- 破解門禁系統
				- 利用隨身碟竊取資料
					- 「空隙攻擊」(air-gap attack)
				- 私接網路對機密主機連線
	- 網路連線安全
		- 機密性、完整性與不可否認性
		- 透過 加密與數位簽章 等技術來實現
		- 網路連線安全的核心概念與實現方式
			- 傳輸加密
				- 應優先採用 通道加密(如 VPN) 與傳輸層加密(如 HTTPS、SSL/TLS 1.2/1.3 )
				- 為避免已知的資安漏洞，應立即停用以下協定
					- SSLv3
						- POODLE 是先把安全連線降級到 SSL 3.0，再利用 CBC 填充驗證缺陷，逐步解出 Cookie 等敏感資料
					- TLS 1.0 / TLS 1.1
			- 資訊加密
				- 對稱式加密演算法
					- 使用 相同的金鑰 進行 加密及解密，效率高
					- 如AES-GCM (AEAD)、ChaCha20-Poly1305 (AEAD)
						- **AES、ChaCha20**：負責「加密」
						- **GCM、Poly1305**：負責「完整性驗證與身分驗證」
						- AEAD
							- Authenticated Encryption with Associated Data 具關聯資料的認證加密
							- 可以保護一段「不加密、但要防竄改」的資料，稱為 **Associated Data，關聯資料**
							- eg
								- 封包標頭(Associated Data，關聯資料)：不加密，但要驗證
								- 封包內容：加密，也要驗證
				- 非對稱式加密演算法
					- 使用一對 公開金鑰及私有金鑰，常用於 金鑰交換及數位簽章
					- 如 RSA 至少使用 2048-bit 或更高長度、P-256(ECC)
				- 為避免已知的資安漏洞，應立即停用以下演算法
					- RC4
						- 存在偏向性漏洞
					- 3DES
						- 因 SWEET32 攻擊的潛在風險
			- 資訊完整性的實現:數位簽章技術
				- 完整性、不可否認性
			- HTTP、FTP 與 TELNET 不安全的連線協定
				- 明文傳輸的風險
			- HTTPS、FTPS 與 SSH 安全的連線協定
				- 加密傳輸的保障
	- 虛擬私有網路 (VPN)
		- VPN類型
			- 遠端使用者存取VPN
				- 一位遠端使用者 (Remote User) 身處廣闊的網際網路 (INTERNET) 上，透過 VPN 設備建立起與公司內部站點 (Site) 的安全連線
				- 輕鬆且安全地存取公司內部的重要資源
					- 檔案伺服器
					- 內部應用程式
					- 其他內部資源
			- Site-to-Site VPN:總部 與 分點之內部網路透過網際網路建立安全通道
				- 每個站點的邊緣(通常是路由器或防火牆)部署 VPN 設備，建立起一條加密的「VPN」隧道
				- 彷彿它們處於同一個大型內部網路中
			- ExtranetVPN:與 合作夥伴間 透過 網際網路 建立安全通道
				- 這種 VPN 允許組織 與 其信任的外部實體 共享特定資源和資訊，同時 保持其他內部數據的隔離和安全
				- 注意存取權限的控管
					- 必須精確地控制 合作夥伴 能夠存取 組織內部哪些資源，以及 哪些內部資源 可以存取 合作夥伴的網路
